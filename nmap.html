1 Chapter 7. Service and Application Version Detection

1.1 Port Scanning

  • 25/tcp: maybe SMTP
  • 80/tcp: maybe HTTP
  • 53/udp: maybe DNS
  • 258/tcp: maybe Checkpoint Firewall-1 GUI or yak Windows chat client
  • so on.

1.2 Version Detection

  • High speed, parallel operation and probe/match definition grammar
  • Determine the application name and version number where available
  • Both TCP and UDP, both textual ASCII and packed binary
  • Multi-platform support
  • If SSL, use OpenSSL (if available), to discover services like HTTPS, POP3S, IMAPS, etc.
  • If SunRPC, launche brute-froce RPC grinder to find number, name and version
  • IPv6 is supported, including TCP, UDP, and SSL over TCP
  • CPE(Common Platform Enumeration) output for interoperation with other software
  • Unknown service fingerprint is printed along with a submission URL
  • Database: more than one thousand service signatures, covering more than 180 unique service protocols from ACAP, AFP, and AIM to XML-RPC, Zebedee, and Zebra

1.3 Usage

  1. Use -sV to enable version detection
  2. Use -A to enable version detection and other advanced features

For example: 

nmap -A -T4 -F www.microsoft.com
# -A: Advanced Options
# -T<0-5>: Set timing template (higher is faster)
# -F: Scan fewer ports than the default scan (only port registered in nmap-service)

To generate a Service Fingerprint: 

nmap -O -Pn -sSV -T4 -d --version-trace -p<port> <target>
# if the service is UDP rather than TCP, substitude -sUV for -sSV

1.4 Technique Described

  1. Port scan, then pass all the open or open|filtered TCP and/or UDP ports to the service scanning module
  2. Those ports are interrogated in parallel
  3. For a single port:
    First of all, see nmap-service-probe File Format
    1. Check to see if the port is one of the ports to be excluded
    2. If the port is TCP, Nmap starts by connecting to it

    3. Once the TCP connection is made, Nmap listens for roughly five seconds

      Many common services, including most FTP, SSH, SMTP, Telnet, POP3, and IMAP service, identify themselves in an initial welcome bannery

      Nmap use regular expression includes substrings which can be used to pick version numbers

      If Nmap gets 'soft match' on the service type, but no version info, it will send probes that are known to recognize the soft-matched service type

    4. Then, Nmap UDP probes start, TCP connections end up here if the NULL probe above fails or soft-matches

      Each probe include a probe string
      Response that come back are compared with a regex

      As with the NULL probe, tests will result in:

      1. a full match (ends processing for the remote service)
      2. a soft match (limits future probes to those which match a certain service)
      3. no match at all

      The exact list of regex depends on the probe fallback1 configuration

      If any response during version detection is ever received from a UDP port which was in the open|filtered state, that state is changed to open

    5. Mostly, the NULL probe or probable port probe matches the service. It's obviously that the NULL probe can be done with other probable port probe, which allows service detection to be done with only one brief connection

      Should the NULL probe and probable port probe fail, Nmap goes through all of the existing probes sequentially, which is quite bad…

      To speed up scans:

      1. Makes most probes generic enough to match many services. 

        1. "\r\n\r\n" # to match many diverse service types
2. GetRequest
3. "help\r\n"
4. RPC and MS SMB probes
      2. If softmatch, only need to try probes that can potentially match the service
      3. Use rarity metric to avoid trying probes that are extremely unlikely to match
    6. Test whether the target port is running SSL. If so, Nmap connects back via SSL and restarts the service scan to determine what is listening behind the encryption
    7. Generic probe identifies RPC-based services
    8. If at least one of the probes elicits some response, Nmap is unable to recognize, the content is printed to user in the form of a fingerprint

1.4.1 Cheats and Fallbacks

To improve performance of NULL probe

Instead of waiting the response of NULL probe, send other probe, which has a fallback to the NULL probe 

1. Give up waiting for NULL probe response of port 25 and send next probe,
   which is "HELP\r\n\r\n". Nmap would match the regex of NULL probe instead
   of HELP probe
2. Instead of GetRequest, use HTTPOptions probe to identify Apache web server
   which has a fallback to the GetRequest regex.

1.4.2 Probe Selection and Rarity

In determining what probes to use, Nmap considers their rarity.
If a probe has a high rarity, it's considered less common and is less likely to be tried.

  1. For TCP, the NULL probe is always tried first
  2. All probes that have the port are tried in the order they appear in nmap-service-probes

  3. All other probes that have a rarity value less than or equal to the current intensity value of the scan are tried, also in the order in nmap-service-probes

    Use this options to change the intensity value of the scan:

    • –version-intensity </intensity level between 0 to 9/>
    • –version-light (intensity level 2)
    • –version-all (intensity level 9!!)

1.5 Post-processors

The post-processors presently available are NSE integration, RPC grinding, and SSL tunneling.

1.5.1 NSE Integration

Regex cannot recognize everything
Some services require custom probe strings or a complex multi-step handshaking process

1.5.2 RPC Grinding

SunRPC (Sun Remote Procedure Call) is a common Unix protocol used to implement many services including NFS
Most RPC service have a high-numbered ports and/or the UDP transport protocol

If the portmapper (rpcbind) service (UDP or TCP port 111) is available, RPC service can be enumerated with the Unix rpcinfo command.

Nmap use the following three-step process to get the RPC information:

  1. TCP and/or UDP port scan finds all of the open ports
  2. Version detection determines which of the open ports use SunRPC protocol
  3. RPC brute force engine determines the program identity of each RPC port by trying a null command against each of the 600 programs numbers in nmap-rpc (Until success is returned for one of them)

1.6 service probe

1.6.1 Exclude Directive

Syntax: Exclude <port specification>

Exclude 53, T:9100, U:30000-40000

Can only be used once and should be near the top of the file, above any Probe directives
Use the same format as the Nmap -p switch

1.6.2 Probe Directive

Syntax: Probe <protocol> <probename> <probestring> 

Probe TCP GetRequest q|GET / HTTP/1.0\r\n\r\n|
Probe UDP DNSStatusRequest q|\0\0\x10\0\0\0\0\0\0\0\0\0|
Probe TCP NULL q|| # listen for the initial banners that many services send
  • protocol
    TCP or UDP
  • probename
    Plain English name for the probe, used to describe which probes elicited responses

  • probestring
    Must start with a 'q', then a delimiter character (like '|', '/') which begins and ends the string

    Standard escape characters: \\, \0, \a, \b, \f, \n, \r, \t, \v, and \xHH

1.6.3 match Directive

Syntax: match <service> <pattern> [<versioninfo>] 

match ftp m/^220.*Welcome to .*Pure-?FTPd (\d\S+\s*)/ p/Pure-FTPd/ v/$1/ cpe:/a:pureftpd:pure-ftpd:$1/
match ssh m/^SSH-([\d.]+)-OpenSSH[_-]([\w.]+)\r?\n/i p/OpenSSH/ v/$2/ i/protocol $1/ cpe:/a:openbsd:openssh:$2/
match mysql m|^\x10\0\0\x01\xff\x13\x04Bad handshake$| p/MySQL/ cpe:/a:mysql:mysql/
match chargen m|@ABCDEFGHIJKLMNOPQRSTUVWXYZ|
match uucp m|^login: login: login: $| p/NetBSD uucpd/ o/NetBSD/ cpe:/o:netbsd:netbsd/a
match printer m|^([\w-_.]+): lpd: Illegal service request\n$| p/lpd/ h/$1/
match afs m|^[\d\D]{28}\s*(OpenAFS)([\d\.]{3}[^\s\0]*)\0| p/$1/ v/$2/
  • service
    service name that the pattern matches

  • pattern

    m/[regex]/[opts]

    regex: Must start with a 'm', then a delimiter character (almost any printable character) which begins and ends the string
    opts: Only options currently supported are 'i', which makes a match case-intensitve and 's' which includes newlines in the '.' specifier

  • versioninfo

    Table 1: versioninfo field formats and values
    Field format Value description
    p/vendorproductname/ Vendor and often service name
    v/version/ Version number
    i/info/ Miscellaneous further information which might be useful
    h/hostname/ Hostname offered up by a service
    o/operatingsystem/ The operating system the service is running on
    d/devicetype/ The type of device the service is running on
    cpe:/cpename/[a] CPE name for some aspect of the service
  • helper function
    • $P(): filter out unprintable characters
    • $SUBST(): make substitutions in matches before they are printed

1.6.4 softmatch Directive

Syntax: softmatch <service> <pattern> 

softmatch ftp m/^220 [-.\w ]+ftp.*\r\n$/i
softmatch smtp m|^220 [-.\w ]+SMTP.*\r\n|
softmatch pop3 m|^\+OK [-\[\]\(\)!,/+:<>@.\w ]+\r\n$|

Main difference is that scanning continues after a softmatch, but it is limited to probes that are known to match the given service

1.6.5 ports and sslports Directives

Syntax: 

ports <portlist>
sslports <portlist>

ports 21,43,110,113,199,505,540,1248,5432,30444
ports 111,4045,32750-32810,38978

1.6.6 totalwaitms Directive

Syntax: totalwaitms <milliseconds> 

totalwaitms 6000

1.6.7 tcpwrappedms Directive

Syntax: tcpwrappedms <milliseconds> 

tcpwrappedms 3000

Only used for the Null probe

If a service closes the TCP connection before this timer runs out, then the service is labeled tcpwrapped

1.6.8 rarity Directive

Syntax: rarity <value between 1 and 9> 

rarity 6

The rarity directive roughly corresponds to how infrequently this probe can be expected to return useful results

1.6.9 fallback Directive

Syntax: fallback <Comma separated list of probes> 

fallback GetRequest, GenericLines

Specifies which probes should be used as fallbacks for if there are no matches in the current Probe section

1.8 Hack Version Detection to Suit Custom Needs, such as Open Proxy Detection

  1. Problem:

    To test an application if they are actually vulnerable, is to exploit it.
    While it's not the problem!

  2. Solution:

    # copy the nmap-service-probes file
mkdir ~/proxydetect
cp /usr/local/share/nmap/nmap-service-probes ~/proxydetect
# force Nmap to use temporary file
export NMAPDIR=$HOME/proxydetect
# Add a probe and match line to probe file,
# just after all the match lines in the NULL probe
# Nmap will try to request an HTTP download by treating any scanned ports as proxies!!
...
Probe TCP ProxyProbe q|GET http://insecure.org/ HTTP/1.1\r\nHost: insecure.org\r\n\r\n|
rarity 1
ports 1-65535
totalwaitms 20000
match proxy m|^HTTP/1.[01] 200 OK\r?\n.*TITLE>Insecure.O|s p/Open HTTP Proxy!!/
...
# If a port is available for a proxy service, it will return "200 OK"

    Many other protocols can be proxies in addition to HTTP.

2 Chapter 8. Remote OS Detection

Two systems for OS Detection:

  1. A huge database of heuristics for identifying thousands of different systems based on how they respond to a selection of TCP/IP probes
  2. Another system (part of version detection) interrogates open TCP or UDP ports to determine device type and OS details

2.1 Usage and Examples

  1. OS detection:

    nmap -O -v <address>
# -O for OS detection, -v for more OS-related details

    -O -v options caused Nmap to generate:

    1. Device type
      Maybe several device types
    2. Running
      It shows the OS Family (such as Linux) and OS generation (2.6.X) if available
    3. OS CPE
    4. OS details
      Contains free-form data which is useful to a human reading the report
    5. Uptime guess
      Receives several SYN/ACK TCP packets in a row and checks the headers for a timestamp option. Simple linear extrapolation determines boot time (it's quite inaccurate)
    6. Network Distance Hops between user and the target host

    7. TCP Sequence Prediction Poor TCP initial sequence number generation will be vulnerable to blind TCP spoofing attacks

      Only printed in verbose mode

    8. IP ID sequence generation Many systems unwittingly give away sensitive information about their traffic levels based on how they generate the lowly 16-bit ID field in IP packets

      Only printed in verbose mode

  2. Version scan to detect the OS Many apps divulge their platform in overly verbose banner messages

OS detection is far more effective if at least one open and one close TCP port are found

  1. With --osscan-limit option Nmap will not even try OS detection against hosts which don't meet the criteria above
  2. With --osscan-guess option Nmap will guess more aggressively
  3. If fail to find a perfect match, it usually repeats the attempt
    (By default, five times if conditions are favorable, and twice if not)
  4. Use numerous reconnaissance methods to explore a network,
    and don't trust any one of them!

2.2 TCP/IP Fingerprinting

Nmap OS fingerprinting works by sending up to 16 TCP, UDP, and ICMP probes to known open and closed ports of the target machine

Details of Nmap OS detection:

  1. Sequence generation (SEQ, OPS, WIN, and T1)
    Six TCP SYN packets to an open port, each is sent 100 milliseconds apart, total 500ms (designed on purpose to detect the common 2Hz TCP timestamp sequences)

    • Packet #1:
      window scale (10), NOP, MSS (1460), timestamp (TSval: 0xFFFFFFFF; TSecr: 0), SACK permitted. The window field is 1
    • Packet #2:
      MSS (1400), window scale (0), SACK permitted, timestamp (TSval: 0xFFFFFFFF; TSecr: 0), EOL. The window field is 63
    • Packet #3:
      Timestamp (TSval: 0xFFFFFFFF; TSecr: 0), NOP, NOP, window scale (5), NOP, MSS (640). The window field is 4
    • Packet #4:
      SACK permitted, Timestamp (TSval: 0xFFFFFFFF; TSecr: 0), window scale (10), EOL. The window field is 4
    • Packet #5:
      MSS (536), SACK permitted, Timestamp (TSval: 0xFFFFFFFF; TSecr: 0), window scale (10), EOL. The window field is 16
    • Packet #6:
      MSS (265), SACK permitted, Timestamp (TSval: 0xFFFFFFFF; TSecr: 0). The window field is 512

    The results of these tests include four result category lines

    1. SEQ: contains results based on sequence analysis of the probe packets 

      names: GCD, SP, ISR, TI, II, TS, and SS

    2. OPS: contains TCP options 

      names: 01-06

    3. WIN: contains window sizes 

      names: W1-W6

    4. T1: contains various test values for packet #1 (almost always the same for each probe) 

      names: R, DF, T, TG, W, S, A, F, O, RD, and Q

  2. ICMP echo (IE) Two ICMP echo request packets:

    1. First:
      IP DF bit set, TOS (type-of-service) byte (0), code of nine, sequence number (295), random IP ID and ICMP request identifier, data payload (120 bytes of 0x00)
    2. Second:
      similar to the first, except a TOS of four (IPTOSRELIABILITY), code of zero, 150 bytes of data (so need a proper payload), the ICMP request ID and sequence numbers are incremented by one

    The results of these tests are conbined into a IE line 

    names: R, DFI, T, TG, and CD
R: only true (Y) if both probes elicit responses
DFI: custom test for this special dual-probe ICMP case
T and CD are for the response to the first probe only

  3. TCP explicit congestion notification (ECN)

    ECN is a method for improving Internet performance by allowing routers to signal congestion problems before they start having to drop packets

    Nmap test this by sending a SYN packet which also has the ECN CWR and ECE congestion control flags set.
    TCP options are WScale (10), NOP, MSS (1460), SACK permitted, NOP, NOP
    The probe is sent to an open port

  4. TCP (T2-T7) Six tests each send one TCP probe packet.

    TCP options data in each case is 0x03030A0102040109080AFFFFFFFF000000000402
    20 bytes data correspond to windows scale (10), NOP, MSS (265), timestamp (TSval: 0xFFFFFFFF; TSecr: 0), then SACK permitted
    Only exception is that T7 uses a window scale value of 15

    The variable characteristics of each probe:

    • T2 sends a TCP null (no flags set) packet with the IP DF bit set and a window field of 128 to an open port
    • T3 sends a TCP packet with the SYN, FIN, URG, and PSH flags set and a window field of 256 to an open port. The IP DF bit is not set
    • T4 sends a TCP ACK packet with IP DF and a window field of 1024 to an open port
    • T5 sends a TCP SYN packet without IP DF and a window field of 31337 to a closed port
    • T6 sends a TCP ACK packet with IP DF and a window field of 32768 to a closed port
    • T7 sends a TCP packet with the FIN, PSH, and URG flags set and a window field of 65535 to a closed port. The IP DF bit is not set

    In each of these cases, a line is added to the fingerprint with results of 

    R, DF, T, TG, W, S, A, F, O, RD, and Q

  5. UDP (U1) A UDP packet sent to a closed port. The character 'C' (0x43) is repeated 300 times for the data field. The IP ID value is set to 0x1042 for operating systems which allow us to set this

    If the port is truly closed and there is no firewall in place, Nmap expects to receive an ICMP port unreachable message in return
    That response is then subjected to the R, DF, T, TG, IPL, UN, RIPL, RID, RIPCK, RUCK, and RUD tests

2.6 OS Matching Algorithms

2.6.1 IPv4 matching

It takes a subject fingerprint and tests it against every single reference fingerprint in nmap-os-db

  1. Looks at each probe category line from the subject fingerprint in turn. Probe lines which don't exist in the reference fingerprint are skipped, until the reference fingerprint have a matching line, they are compared

  2. When match a test, Nmap increments the PossiblePoints accumulator by the numbers of points assigned to this test
    Then compare the value, if matches, the NumMatchPoints accumulator is incremented by the test's point value

    Test point values are assigned by a special MatchPoints entry

  3. Once all of the probe lines are tested for a fingerprint, Nmap devides NumMatchPoints by PossiblePoints. The result is a confidence factor

  4. Once all of the reference fingerprints have been evaluated, Nmap orders them and prints the perfect matches (if not too many), if there are no perfect matches, but some are very close, it also print those.

    Guesses are more likely to be printed if the --osscan-guess option is given
